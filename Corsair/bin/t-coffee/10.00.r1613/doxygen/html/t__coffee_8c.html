Remark

The follwowing text is just a summation of short lines of documentation appearing throughout the code of this function. Often they make more sense in their specific context, that means while reading the source code.

Preparation

Before anything else, check if we want to use the other_pg option of T-Coffee. If so, redirect to run_other_pg. (In case of kmcoffee, load the kmcoffee arguments from km_coffee)

Read all the parameters of T_Coffee using get_cl_param

Load T-Coffee default parameters. Next, decide whether a special mode of T_Coffee has been called and load the corresponing parameters from a function. Here's an example: 

 else if ( strm (special_mode, "psicoffee")) new_arg = get_psicoffee_defaults( NULL,lseq_type );

Special commands, that usually end T-Coffee are handled next: check_configuation, update, version etc.

If T-Coffee is invoked without any arguments, it displays the available method names through display_method_names and exits.

Array list_file[] contains the names of all input files like sequences, profiles and templates, but also methods. There is a convention in T-Coffee to start the filename of with a capital letter specifying its type:

  • M for methods
  • R for profiles
  • A for alignments
  • L for libraries (via the -lib flag)
  • P for PDB structures

See read_constraint_list for more information on how these files will be processed.

If no alignment A, library L or method M is in the list of infiles (that means in list_files[]), T-Coffee will add a default method here. The default method is currently Mproba_pair.

Check the content of each input file and report possible errors.

First non-error output: List the Input files, which are usually a sequence file (S) and one or several methods (M), as shown in the example: 

 INPUT FILES
         Input File (S) sh3.fasta  Format fasta_seq
         Input File (M) proba_pair

Input Sequences and Libraries

Read the Sequences (read_seq_in_n_list). Moreover, for these methods that need Blast scans on the sequences, precompute_blast_db is called.

Note:
The precompute_blast_db method is currently invoked only for methods that contain the substring "blast". As far as I know, it's main purpose is the method blastr_pair which is used in the special mode blastr. See the source code of precompute_blast_db for mmore information.

Translate Sequences from RNA to DNA or vice versa, if wanted (transform_sequence).

Reorder the sequences, if wanted, by using reorder_seq. You can make T-Coffee order the sequences lexicographically by their names by using the -inorder=aligned flag. If it is set, the order of the sequences is specified by sort_string_array(new_oder), otherwise the order from the input file (i.e. the order in the Sequence::name array) is kept.

Prepare the global Constraint_list object using declare_constraint_list. This function allocates memory for a Constraint_list and sets the Constraint_list::residue_index to default values.

Identify Master Sequences for the master mode (option -master). Per default, master mode is not used (-master=no), but there are several ways to specify master sequences. The master mode is experimental only, the idea is not to run all pairwise alignments, but only some of them. See prepare_master (best in the soruce code) for more details.

Get the Templates

Call seq2template_seq for each template file in template_file_list. This will read/fetch template files and incorporate them into the Seqeunce object. If no template file is given but the Sequence object already contains templates, these are written into a .template_file file. Next,the same procedure is repeated for profile_template_files (-profile_template_file), using profile_seq2template_seq

See also:
Template for more information on how templates are stored within T-Coffee.

Add the sequences as their own template via the generic seq2template_seq and the option "SELF_S". Finally, specify the types of used templates in the Sequence object by seq2template_type.

Set further options like the potential substitution matrix, the -filter_lib flag, the profile_mode (see get_profile_mode_function), some BLAST parameters, some PDB-BLAST parameters and the multicore mode, before we can go on to the real stuff.

Compile the Library

Build Up the Initial Constraint_list

Finally we can start with the actual algorithm to build an alignment. In this step we build the initial Constraint_list from various sources. Usually, the initial Constraint_list will contain the information from pairwise alignment methods (per default proba_pair), but of course there are different modes of T-Coffee that use structural pairwise alignments or the output of multiple aligners. However, this is the place where all this information is produced and collected. The function that summarizes all this is read_n_constraint_list and its documentation is recommended reading.

Just to give a short overview: It will distibute the process of generating pairwise alignments (i.e. calling external aligners or built-ins like proba_pair) over several CPUs and incorporate their outputs in one single Constraint_list object called CL. There are some maybe non-intuitive decisions done in the process of reading the outputs, which should be documented in the function.

See also:
Constraint_list::residue_index to understand how CL stores edges/constraints.

Some methods, like proba_pair, need a cleanup afterwards to free static memory allocated during their process. That's why free_pair_wise is called here.

If the Constraint_list should, for some reason, be empty afterwards, report an Error.

If an -extend has been set to a value greater than 0 an initial filtering of the Constraint_list is performed, see filter_constraint_list.

Choose the extension mode given by the -extend flag, see choose_extension_mode. Default values seems to be very_fast_triplet (??)

See also:
Constraint_list::evaluate_residue_pair for more about the Extension mode.

Consistency: Relaxation, Filtering, Extension!

This is where the magic happens!

First, if -out_lib is used, the library will be written into the specified file via the save_constraint_list function. If you demand an extended library (by -out_lib_mode = extended) then first the extension steps are performed and then the library is written to a file via save_extended_constraint_list. If this was all you wanted to do (lib_only), exit the program afterwards.

Extension step:

  1. If filter_lib > 0 then reduce the Constraint_list::residue_index by removing edges with weight smaller or equal to filter_lib. Usually not used right here. See filter_constraint_list.
  2. Only if the environment variable EXTEND4TC is set to 1, call the extend_constraint_list.
  3. For a=0 to relax_lib (which is 1 per default) call relax_constraint_list.
  4. For a=0 to shrink_lib call shrink_constraint_list, but schrink_lib is usually set to 0.
  5. A call to evaluate_constraint_list_reference determines the maximum edge weights.

We recommend reading the documentation of each of these functions to get to grips with the extension.

If weight is not set to no (but to sim or matrix for example), then compute a new distance matrix (cl2distance_matrix) and weight_constraint_list with the seq_weight flag.

Quadruplet mode (experimental). Only if nseq_for_quadruplet > 0.

Progressive Alignment

At first, transform the Seqeunce into an Alignment via the small function seq2aln and use ungap_array to remove unwanted gaps. Then you have to choose the right msa_mode for evaluating columns. Right now, there are several options in the code, but only a few of them are used in general.

  • sorted_aln
  • seq_aln
